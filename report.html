1. Problem Statement

1.1 The Problem

Managing shared funds in traditional finance requires trusting a single custodian — a bank, a treasurer, or an individual — who can unilaterally move money. This creates a single point of failure: if that person is compromised, hacked, or acts maliciously, funds are lost with no recourse.

In decentralised systems, the problem is worse. A single private key controls an Ethereum address. If that key is stolen or lost, all funds are gone permanently — no bank to call, no chargeback, no recovery. For DAOs, investment clubs, corporate treasuries, or any multi-party arrangement, this is unacceptable.

1.2 Why Blockchain is the Right Tool

  • Trustless execution: Smart contracts enforce the M-of-N rule in code — no single party can override it, not even the contract deployer.
  • Immutable audit trail: Every submission, approval, revocation and execution is recorded on-chain via events. The full history is publicly verifiable.
  • No intermediary: No bank, escrow agent, or lawyer is needed. The contract is the agreement.
  • Programmable: The wallet executes arbitrary calldata — not just ETH transfers but any on-chain function call — enabling DAO governance, protocol interactions, and asset management.
  • Composability: Any DeFi protocol, NFT contract, or DAO can be the target of a multi-sig transaction, making this a universal governance primitive.
Real-world relevance: Gnosis Safe (our architecture inspiration) secures over $50 billion in assets for DAOs, protocols, and institutions — proving multi-sig is the industry standard for shared on-chain treasury management.

1.3 Our Solution

We implement a configurable M-of-N Multi-Signature Wallet on Ethereum. Any N owners are defined at deployment; any transaction requires at least M approvals before it executes. Owners can submit transactions, approve, revoke, and execute — all enforced atomically on-chain with no trusted third party.

3. Security Analysis

3.1 Reentrancy Attack — Prevented

The most critical vulnerability in ETH-sending contracts is reentrancy. An attacker contract's receive() can call back into executeTransaction() before state is updated, draining funds multiple times.

Double protection applied:

  • Checks-Effects-Interactions (CEI): txn.executed = true is set before the external .call(). Any reentry attempt hits the notExecuted modifier and reverts immediately.
  • OpenZeppelin ReentrancyGuard: The nonReentrant modifier on executeTransaction uses a mutex lock that blocks all reentrant calls at the EVM level.
// CEI Pattern — Effects before Interactions function executeTransaction(uint256 _txId) external onlyOwner txExists(_txId) notExecuted(_txId) nonReentrant { Transaction storage txn = transactions[_txId]; if (txn.approvalCount < requiredApprovals) revert InsufficientApprovals(); txn.executed = true; // ← EFFECT first (bool success, ) = txn.to.call{value: txn.value}(txn.data); // ← INTERACTION second if (!success) revert TransactionFailed(); }
✅ Tested with ReentrancyAttacker.sol — the attacker's receive() attempts re-entry, receives only 1 ETH, and the wallet balance is intact.

3.2 Access Control — Prevented

All state-changing functions are protected by the onlyOwner modifier:

modifier onlyOwner() { if (!isOwner[msg.sender]) revert NotOwner(); _; }

The isOwner mapping provides O(1) lookup. Non-owners calling any function receive a NotOwner revert — verified by tests for all 4 state-changing functions.

3.3 Replay / Double-Execution — Prevented

Once a transaction is executed, txn.executed = true is permanent. The notExecuted modifier blocks any subsequent call to approve, revoke, or execute the same transaction.
